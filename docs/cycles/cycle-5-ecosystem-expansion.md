# 周期 5 开发文档 — 多端与生态扩展

> **项目**: My Application — Android 音乐播放器  
> **周期定位**: 周期 5 / 建议 2 周  
> **对应总文档**: `docs/technical-implementation.md`

---

## 1. 周期目标

本周期开始从“播放器”扩展到“多端入口 + 内容生态外壳”，把 Widget、驾驶模式、Android Auto、MV 和社区能力逐步铺开。

---

## 2. 范围定义

### 2.1 前端范围

| 模块 | 任务 | 结果 |
|------|------|------|
| 桌面入口 | 桌面小组件 | 支持歌曲信息展示和基础播放控制 |
| 大屏场景 | 驾驶模式 | 支持大按钮、大字体、简化交互 |
| 车机场景 | Android Auto | 支持媒体浏览树和播放控制 |
| 视频内容 | MV / 视频播放 | 支持进入视频播放页和全屏播放 |
| 社区内容 | 动态 / 社区外壳 | 先完成列表、详情和分页骨架 |

### 2.2 后端范围

| 模块 | 任务 | 结果 |
|------|------|------|
| 视频能力 | `getMvUrl` | 返回 MV 可播放地址和必要元数据 |
| 社区能力 | Feed / 详情 / 评论接口 | 支撑社区列表和详情页 |
| 多端数据 | 媒体元数据完整性 | 保证 Widget / Auto / 视频页所需字段完整 |

## 3. 前端开发清单

### 3.1 桌面小组件

**涉及文件**

- `widget/MusicWidgetProvider.kt`
- `widget/MusicWidgetReceiver.kt`
- `res/xml/music_widget_info.xml`

**开发要求**

- 优先采用 `Glance`
- 支持显示封面、标题、歌手和基础控制按钮
- 播放状态变化时及时刷新

**第 1 周前半完成项**

- 首版规格先锁定为一个稳定的小组件规格，不并行开多尺寸，先把刷新链路和交互链路干扎实
- 小组件展示信息统一收敛为：封面、歌曲名、歌手、播放态；首版不塞收藏、音质、歌词等花活
- 控制能力收敛为：打开应用、播放 / 暂停、上一首、下一首，避免把交互面铺得太大
- 刷新触发点明确为：曲目切换、播放状态切换、系统恢复小组件实例；首版不做高频进度条刷新
- 空态统一展示应用名、默认封面和“打开播放器”入口，避免无播放内容时留白发呆

**当前实现约定**

- 小组件状态源以当前播放快照为准，不单独维护第二套播放器状态，别把状态中心拆成两摊烂账
- 小组件消费字段只依赖 `PlaybackState` + `Track` 现有能力：`title`、`artist`、`coverUrl`、`isPlaying`
- 如果封面获取失败，默认回退到纯色占位和文本展示，不能因为图片拉跨把整个 Widget 干废
- 首版验收以“状态正确、按钮可用、刷新稳定”为准，不把视觉特效当主要目标

### 3.2 驾驶模式

**涉及文件**

- `feature/driving/DrivingModeScreen.kt`
- `feature/driving/DrivingModeViewModel.kt`

**开发要求**

- 复用现有播放控制，不另造控制链
- UI 以大按钮、大标题为主
- 可选预留语音控制入口，但本周期不强求完整语音方案

**第 1 周前半完成项**

- 驾驶模式先按“独立页面骨架”收口，不塞进底部主导航，避免主导航先被功能试验田挤爆
- 页面结构收敛为：当前歌曲信息、大封面 / 占位区、超大播放控制区、简化进度区
- 页面状态直接复用全局 `PlayerViewModel` 的播放态和控制能力，不新增播放服务分支
- 首版目标是“大按钮 + 大字号 + 少决策”，不做歌词滚动、评论、音效等高干扰内容
- 语音入口仅保留占位说明和后续扩展点，第 1 周前半不承诺语音链路落地

**当前实现约定**

- 驾驶模式只允许保留核心操作：播放 / 暂停、上一首、下一首、退出；复杂功能全部后置
- 交互优先级以可点中、可一眼看清为先，视觉花哨度往后稍稍，别把安全场景做成秀场
- 页面需要兼容“无当前播放内容”的空态，避免用户点进来只看见一坨空白
- 首版支持竖屏主布局即可，横屏和车机适配在 Android Auto 阶段继续补齐

### 3.3 Android Auto

**涉及文件**

- `MusicPlaybackService.kt`
- `AndroidManifest.xml`
- `automotive_app_desc.xml`

**开发要求**

- `MusicPlaybackService` 需具备 `MediaLibraryService` 能力
- 返回基础媒体浏览树：收藏、最近播放、歌单
- 需要真机或模拟器验证

### 3.4 MV / 视频播放

**涉及文件**

- `feature/player/VideoPlayerScreen.kt`
- `media/video/VideoPlayerManager.kt`

**开发要求**

- 支持从播放页更多菜单进入
- 支持全屏、横屏和退出回到音频播放页
- 视频播放异常不能拖垮主音频服务

### 3.5 动态 / 社区外壳

**涉及模块**

- `feature/community` 预留模块

**开发要求**

- 先完成 Feed 列表、详情、空态、错误态
- 评论、点赞、分享等交互是否在本周期完整落地，取决于后端接口成熟度

## 4. 后端开发清单

### 4.1 MV 接口

**接口要求**

- `getMvUrl(platform, songId)` 或等价能力
- 返回视频播放地址、封面、清晰度信息、时长等基础字段

**第 1 周前半接口契约**

> 客户端先按下面这套字段契约对齐，服务端具体 URL 命名可以等价调整，但字段语义别乱飞。

| 项目 | 约定 |
|------|------|
| 接口能力 | `getMvUrl(platform, songId)` 或等价 HTTP 接口 |
| 请求字段 | `platform`、`songId`、`mvId?`、`preferredQuality?` |
| 成功返回 | `mvId`、`title`、`artist`、`coverUrl`、`durationMs`、`playUrl`、`qualities[]`、`sourcePlatform` |
| 质量字段 | `qualities[]` 至少包含 `code`、`label`、`bitrate?`、`resolution?`、`isDefault` |
| 时效字段 | 如播放地址带签名，需返回 `expiresAt`，前端据此决定是否重新解析 |
| 空结果约定 | 无 MV 时返回明确业务空态，不要用模糊异常把前端忽悠瘸了 |
| 错误语义 | 参数错误、平台不支持、上游失败要能区分，别所有失败都糊成 `500` |

**当前实现约定**

- `songId` 是最小必填字段；若平台侧存在独立 `mvId`，服务端返回后前端后续调用优先使用 `mvId`
- `playUrl` 必须是客户端可直接交给播放器使用的地址，别再让前端套娃二次解析
- 视频字段不强塞进现有 `Track`，而是定义独立的 MV / Video DTO，省得把音频模型拧成麻花
- 首版只要求单 MV 可播放链路，相关推荐、弹幕、多清晰度手动切换都先不进第一周前半范围

### 4.2 社区接口

**接口要求**

- Feed 列表
- 动态详情
- 评论列表
- 分页参数与排序约定
- 鉴权和匿名访问策略

**第 1 周前半接口契约**

| 能力 | 最低契约 |
|------|----------|
| Feed 列表 | 返回 `items`、`nextCursor`、`hasMore`、`serverTime`，满足只读列表滚动 |
| 动态详情 | 返回正文块、作者信息、发布时间、媒体附件、统计字段、关联歌曲 / 歌单信息 |
| 评论列表 | 返回 `items`、`nextCursor`、`hasMore`、`sort`，至少支持 `hot` / `latest` |
| 分页策略 | 优先使用 `cursor`，保留 `page` / `pageSize` 兼容能力，但别两套语义打架 |
| 排序策略 | Feed 至少支持 `latest`；评论至少支持 `hot`、`latest` |
| 匿名策略 | 第 1 周前半只读浏览允许匿名；点赞、评论、分享等写操作统一后置 |
| 鉴权反馈 | 未登录时返回 `interactionEnabled = false` 或等价字段，前端据此禁用交互入口 |

**社区最小字段模型**

| 模型 | 必需字段 |
|------|----------|
| FeedItem | `postId`、`author`、`summary`、`publishTime`、`media[]`、`commentCount`、`likeCount` |
| FeedDetail | `postId`、`author`、`contentBlocks[]`、`publishTime`、`media[]`、`topicTags[]`、`relatedTracks[]` |
| CommentItem | `commentId`、`author`、`content`、`publishTime`、`likeCount`、`replyCount` |

**当前实现约定**

- 社区模块首版只做只读浏览闭环，不在第一周前半硬塞点赞、评论发布、分享跳转这些高变更项
- 社区内容使用独立模型体系，和 `Track` 仅保留弱关联，例如 `relatedTrackId`、`relatedPlaylistId`
- 富文本统一先收敛为文本块 + 图片块的组合，别第一把就把富文本编辑器和全媒体协议扛上来
- 所有列表接口都必须返回明确空态语义，前端才能把空态和错误态分开，不至于全都显示“网络异常”

### 4.3 多端数据支持

**需要确认**

- Widget / Auto 所需的封面、标题、时长、播放状态字段是否完整
- 社区富文本或多媒体字段是否需要额外约束

**第 1 周前半字段盘点结论**

| 消费场景 | 关键字段 | 当前结论 |
|------|------|------|
| Widget | `title`、`artist`、`coverUrl`、`isPlaying` | 基于现有 `Track` + `PlaybackState` 已够首版使用 |
| 驾驶模式 | `title`、`artist`、`coverUrl`、`durationMs`、`positionMs`、`isPlaying` | 现有播放状态可直接复用，无需额外补模型 |
| Android Auto | `queue`、`currentIndex`、`title`、`artist`、`album`、`coverUrl` | 现有字段基本够用，后续重点在 `MediaLibraryService` 输出组织 |
| MV 页面 | `mvId`、`playUrl`、`coverUrl`、`durationMs`、`qualities[]` | 现有 `Track` 不够，需新增独立视频 DTO |
| 社区 | `postId`、`author`、`summary`、`contentBlocks[]`、`media[]` | 应独立建模，不建议把社区字段塞进播放模型 |

**当前结论**

- 多端播放展示依赖的基础字段已经够用，第一周前半不需要为了 Widget / 驾驶模式去硬改 `Track`
- MV 和社区都是“新增内容模型”问题，不是给现有音频模型缝补几个字段就能糊过去
- 后续如果要做 Widget 深链、Auto 浏览树或社区关联跳转，优先新增轻量级路由参数，不改核心播放状态结构
